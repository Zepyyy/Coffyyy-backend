# Coffyyy Backend

NestJS/Prisma API backing the Coffyyy frontend's local-first-to-cloud sync migration (frontend issue #10, umbrella; #18 outbox push, #19 pull/reconciliation).

## Language

**Client ID**:
A UUID generated by the frontend when a record is created offline, before the server has assigned identity. Sent on push as a dedup/idempotency key.
_Avoid_: local ID, temp ID

**Server ID**:
The existing `Int @default(autoincrement())` primary key on Bean/Brew/Machine. Stays the real PK; unaffected by sync work.
_Avoid_: record ID (ambiguous with Client ID)

**Revision**:
A single monotonic counter per User (workspace), shared across Bean/Brew/Machine. Every create/update/delete on any of that user's rows stamps it with the next value. Pull uses "revision > N" as the change-discovery cursor; LWW compares revisions within a record.
_Avoid_: version, timestamp (LWW must never use device/wall-clock time)

**Change**:
An append-only log entry (userId, entityType, Server ID, Client ID, Revision, operation, payload snapshot) recording one create/update/delete. Source of truth for pull, tombstones (operation = delete), and 7-day recoverable losing-version history. Purged after 7 days.
_Avoid_: audit log, tombstone table (tombstones are just Changes with operation = delete, not a separate structure)

**Push Operation**:
A dedicated dedup record (userId, operationId, payloadHash, result, revision) for one outbox push, distinct from `ImportRun` (which stays scoped to the one-time migration import flow).

## Decisions

- Deleting a Bean or Machine never hard-deletes the row or cascades to its Brews. The row stays in place (soft-delete signaled via a Change entry); `Brew.beanId`/`machineId` stay required and never dangle, so historical brews keep full context on a "deleted" bean/machine.
